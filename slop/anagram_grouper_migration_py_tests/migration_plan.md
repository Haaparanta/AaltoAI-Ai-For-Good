# Migration Plan

## Summary
This project is a very small single-module Python utility centered on one public function, `group_anagrams(words: list[str]) -> list[list[str]]`. It groups input strings by the tuple of their sorted characters and returns the grouped words in insertion order by first-seen anagram class. There are no external dependencies, no package structure, and only a simple `__main__` self-check block.

| `source/__pycache__/...` | cache artifact | none | Interpreter-generated bytecode; not part of source behavior. |

### `source/main.py`
- Defines `group_anagrams(words: list[str]) -> list[list[str]]`.
- Implementation uses a dictionary keyed by `tuple(sorted(word))`.
- Group accumulation is done with `groups.setdefault(key, []).append(word)`.
- Returns `list(groups.values())`.
- Script mode (`if __name__ == "__main__":`) runs a few assertions covering common, empty, singleton, duplicate, and all-anagram cases.

## Behavior
- Input is expected to be `list[str]` per annotation and stub.
- For each word, the function computes `tuple(sorted(word))` as the canonical anagram key.
- Words with identical sorted-character tuples are appended to the same output sublist.
- Output ordering behavior is important:
  - groups appear in the order their key is first encountered in the input, because Python dictionaries preserve insertion order;
  - words inside each group preserve original input order, because they are appended as encountered.
- Duplicates are preserved exactly; repeated identical strings are kept as repeated elements in a group.
- Empty input returns `[]`.
- There is no explicit error handling or validation; non-string items would fail when `sorted(word)` is evaluated in ways dependent on the object passed.
- No file I/O, network I/O, environment access, subprocesses, randomness, or concurrency are used.

## Migration risks & mitigations
- **Order-sensitive behavior hidden in implementation details:** `return list(groups.values())` means output depends on dictionary insertion order.  
  **Mitigation:** tests should lock in group order by first occurrence and element order within groups.
- **Canonicalization cost and Unicode/character semantics:** anagram detection is based on Python character sorting, not frequency counts. For simple Rust parity, sorting Unicode scalar values/characters should match intended behavior for ordinary strings.  
  **Mitigation:** include tests for basic ASCII and possibly mixed-case behavior; preserve exact character-sensitive grouping rather than normalizing.
- **No runtime type checks:** Python annotations are informative only. Rust implementation will likely require stricter types.  
  **Mitigation:** define Rust API around `Vec<String>` / `&[String]` (or equivalent) and keep Python-facing bindings aligned if needed.
- **Single-file script structure:** script assertions are not a reusable test suite.  
  **Mitigation:** convert script examples into proper pytest cases.

## Proposed test focus
Tester should lock down these observable behaviors with pytest:
- Standard grouping case: `['eat', 'tea', 'tan', 'ate', 'nat', 'bat']` groups into three anagram classes.
- Empty input returns empty list.
- Singleton input returns a single singleton group.
- Duplicate words are preserved in the output group, not deduplicated.
- All words in one anagram class produce exactly one group.
- Group ordering follows first occurrence of each anagram signature in the input.
- Element ordering within each group follows original encounter order.
- Case sensitivity is preserved (e.g. uppercase/lowercase should not be normalized unless source behavior shows otherwise).
- Non-anagram words with same lengths are kept separate.

## Proposed Rust layout
Given the project size, a minimal Rust shape is appropriate:

```text
rust/
  Cargo.toml
  src/
    lib.rs        # public grouping function
    main.rs       # optional demo / CLI parity with Python script block
```

Suggested mapping:
- `source/main.py::group_anagrams` -> `rust/src/lib.rs::group_anagrams`
- Python `__main__` self-check block -> Rust unit/integration tests, or an optional tiny `main.rs` example if script parity is desired.

Implementation guidance for Translator:
- Represent grouping with an ordered map strategy so output order matches Python first-seen key order. In Rust, a plain `HashMap` alone is not sufficient if deterministic insertion order must be preserved; use either an insertion-order map or maintain a parallel vector of first-seen keys/order.
- Preserve duplicates and original order within each grouped vector.
